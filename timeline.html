Phase 01 · January 2026
Network Restructuring
The foundation everything else is built on. Configured the Raspberry Pi 3B as a Wi-Fi to Ethernet WAN bridge with NAT, DHCP, and firewall isolation rules — creating a fully isolated lab network behind the ASUS router with no access to the home LAN. This mirrors real-world ISP-style network boundary architecture.
Completed
Phase 02 · January 2026
Suricata IDS Deployment
Deployed Suricata on a Raspberry Pi 4 as a Gateway IDS — positioned so all lab traffic flows through it. Configured EVE JSON output for SIEM ingestion, built a custom logrotate policy to protect the SD card, and validated 0 packet drops under full load. This is the network-layer sensor for the entire lab.
Completed
Phase 03 · January 2026
Cowrie Honeypot Deployment
Deployed Cowrie on a second Raspberry Pi 4 as a medium-interaction SSH/Telnet honeypot. Resolved Python version conflicts by compiling Python 3.11 from source, and configured a systemd service for 24/7 persistence. Captures real attacker behavior — brute-force attempts, commands run, and tools used.
Completed
Phase 04 · February 2026
Proxmox Hypervisor Setup
Installed Proxmox VE on a dedicated Mac Mini (i5, 32GB RAM, 512GB SSD) as the bare-metal virtualization backbone. Configured network bridges, storage pools, and the web management interface. This host runs all lab VMs and is the platform all future detection engineering work is built on.
Completed
Phase 05 · February 2026
Ubuntu SOC VM Deployment
Stood up the central SOC backbone VM on Proxmox. Ubuntu Server 22.04 LTS was chosen for its stability and compatibility with the planned Wazuh + Elastic Stack pipeline. Configured static IP, UFW firewall baseline, and NTP synchronization — the hub all other nodes will eventually report into.
Completed
Phase 04b · April 2026 — Active
Proxmox Reinstall — Storage Reconfiguration
During routine lab expansion, Proxmox's LVM-thin storage pool failed to initialize correctly on the Apple SSD — traced to a 4Kn (4096-byte physical sector) alignment incompatibility with LVM-thin's metadata layer. Resolution: reinstall Proxmox VE selecting the ext4 directory-based storage backend, which has no 4Kn constraints. Architecture, network bridges, and VM configurations are fully designed and validated — this is a storage backend swap, not a design change. All VMs will be redeployed against the same spec once the host is restored.
Active
Phase 05b · April 2026 — Pending Rebuild
Ubuntu SOC VM Rebuild (VM 100)
Ubuntu Server 22.04 LTS SOC backbone (VM 100) will be redeployed immediately following the Proxmox reinstall. UFW baseline, static IP assignment, NTP synchronization, and firewall rules are documented and will be applied from the validated configuration — redeployment is procedural, not exploratory.

Phase 07 · Q2 2026 — Pending Install
Windows Server 2022 + Active Directory DC · DNS · DHCP (VM 102)
VM 102 will be deployed as the lab domain controller for lab.local, serving as DC, integrated DNS authoritative server, and DHCP server for the SOC subnet — consolidating identity and network services under a single managed node consistent with enterprise AD architecture. Audit policy configuration and event forwarding to Wazuh are fully planned. Deployment follows Windows 11 rebuild confirmation.
Pending Install
Phase 08 · Q2 2026
Kali Linux Deployment
Deploy Kali Linux as the attacker VM inside Proxmox once Active Directory is live. Sequenced here deliberately — the most valuable attack scenarios (Kerberoasting, Pass-the-Hash, lateral movement) require a domain environment. This phase activates the full attacker-defender loop and makes every upstream detection component meaningful.
Planned
Phase 09 · Q2 2026
Wazuh Manager Integration
Will deploy Wazuh Manager on the Ubuntu SOC VM and install agents across all Windows VMs and Raspberry Pi sensors. Wazuh provides the rule engine that transforms raw logs into structured, actionable alerts — the core of the detection pipeline.
Planned
Phase 10 · Q2 2026
Elastic Stack + SIEM Dashboard
Will integrate Elasticsearch, Logstash, and Kibana for log storage, normalization, and visualization. Kibana dashboards will surface detection alerts and provide analyst workflows — demonstrating the full SIEM pipeline from raw endpoint telemetry to actionable alert.
Planned
Phase 11 · Early Q3 2026
Detection Rule Authoring & MITRE ATT&CK Mapping
Write custom Wazuh detection rules and decoders mapped to MITRE ATT&CK technique IDs. Build a detection coverage matrix documenting what is detected, what is missing, and why. Validate each rule against real attack simulations from Kali. Each rule documented with its technique ID, logic, true-positive test case, and false-positive suppression.
Planned
Future Evolution · Post Q3 2026
VLAN Segmentation & Enhanced Network Visibility
Introduce enterprise-grade network separation using VLANs: servers/SIEM, workstations, attacker/research, and Raspberry Pi cluster on dedicated segments. Add inter-VLAN firewall rules and log blocked traffic to create segmentation-based detection scenarios that mirror real corporate network architecture.
