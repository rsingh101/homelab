Proxmox Configuration
✓ Completed Category: Virtualization Updated: March 2026

Purpose

Proxmox VE serves as the bare-metal hypervisor backbone of the homelab. It hosts all virtual machines and provides the centralized management interface for resource allocation, VM lifecycle management, and storage pooling. This layer is the foundation that makes the entire SOC simulation environment portable and reproducible.

Architecture

ComponentDetailStatus
HypervisorProxmox VE (KVM-based)Online
Storage PoolLocal LVM + ZFSOnline
Network Bridgevmbr0 (LAN), vmbr1 (isolated SOC)Online
VM Count4 active VMsRunning

Steps Taken

1. Downloaded Proxmox VE ISO from the official repository and created a bootable USB via Rufus. Installed on dedicated hardware with EFI boot enabled.

2. Configured network bridges to segment lab traffic from the host LAN. Created vmbr1 as an isolated internal bridge for SOC VM communications.

bash # /etc/network/interfaces — Proxmox bridge config auto vmbr0 iface vmbr0 inet static address 192.168.1.100/24 gateway 192.168.1.1 bridge-ports eno1 bridge-stp off bridge-fd 0 auto vmbr1 iface vmbr1 inet static address 10.10.10.1/24 bridge-ports none bridge-stp off bridge-fd 0

3. Enabled Proxmox SPICE and VNC console access for headless VM management. Configured storage pools for ISO storage and VM disk images.

Lessons Learned

Properly planning network bridge topology before deploying VMs saves significant reconfiguration effort. The isolated vmbr1 bridge is critical for ensuring SOC telemetry doesn't leak onto the production LAN. Always allocate more RAM than initially estimated — SIEM workloads are memory intensive.

Proxmox's web UI provides adequate management for this lab scale, but consider Terraform + Proxmox provider for future automated VM provisioning as the lab expands.

Ubuntu Server Setup
✓ Completed Category: SOC Backbone Updated: March 2026

Purpose

The Ubuntu Server VM acts as the central SOC backbone — the aggregation point for telemetry from all other nodes. It is designed to eventually run the full Wazuh + Elastic Stack SIEM pipeline. Currently it serves as the log aggregation and management server.

Architecture

ComponentDetailStatus
OSUbuntu Server 22.04 LTSOnline
vCPUs4 coresRunning
RAM8 GBRunning
Networkvmbr1 (10.10.10.10)Online
Wazuh ManagerPlanned deploymentPlanned

Steps Taken

Deployed Ubuntu Server 22.04 LTS as a Proxmox VM with a static IP on the isolated SOC bridge. Hardened the SSH configuration and configured UFW firewall rules to allow only necessary inbound traffic.

bash # UFW firewall baseline sudo ufw default deny incoming sudo ufw default allow outgoing sudo ufw allow from 10.10.10.0/24 to any port 22 sudo ufw allow 9200/tcp # Elasticsearch sudo ufw allow 1514/tcp # Wazuh agent sudo ufw enable

Configured NTP synchronization across all lab VMs to ensure log timestamp consistency — critical for accurate SIEM correlation.

Lessons Learned

Time synchronization between all lab nodes is not optional — misaligned timestamps make log correlation nearly impossible. NTP should be configured as a first step on every new VM deployment, not an afterthought.

Sysmon Configuration

Sysmon is deployed with the community-maintained SwiftOnSecurity configuration, which provides broad detection coverage while filtering known-good noise. Key event IDs being captured:

powershell # Install Sysmon with config .\Sysmon64.exe -accepteula -i sysmonconfig-export.xml # Verify Sysmon service Get-Service Sysmon64 # Key Event IDs monitored: # EID 1 - Process Creation # EID 3 - Network Connection # EID 11 - File Create # EID 13 - Registry Value Set # EID 22 - DNS Query

Lessons Learned

Sysmon without a properly tuned configuration generates overwhelming noise. Start with the SwiftOnSecurity base config and iterate — suppressing known-good processes first before adding detection rules. The quality of endpoint telemetry directly determines detection fidelity downstream in the SIEM.

Documentation / Network
Network Segmentation & Routing
✓ Completed Category: Infrastructure Updated: March 2026

Purpose

The physical network is entirely wireless — all devices connect through the ASUS router. The lab achieves isolation from the home network via a double-NAT boundary enforced by the Raspberry Pi 3B WAN bridge, not through VLANs or managed switches. This is a deliberate design that demonstrates how strong isolation can be achieved with minimal hardware using sound Linux networking fundamentals.

Full Traffic Flow

topology [ Home Network: 10.0.0.0/24 ] │ │ Wi-Fi upstream (wlan0: 10.0.0.16) ▼ [ Raspberry Pi 3B — WAN Bridge ] │ NAT · DHCP · Firewall isolation │ eth0: 192.168.100.2 → downstream ▼ [ ASUS RT-AX88U — Lab Gateway ] │ WAN: 192.168.100.x (DHCP from Pi) │ LAN: 192.168.50.0/24 │ ├──── Raspberry Pi 4 — Suricata IDS (all traffic inspected) ├──── Raspberry Pi 4 — Cowrie Honeypot └──── Proxmox Server (Mac Mini — wired for stability) │ vmbr0 → 192.168.50.x (ASUS LAN) ├── Ubuntu SOC VM (Wazuh + Elastic — planned) ├── Windows 11 VM (Sysmon + Wazuh agent — in progress) ├── Kali Linux VM (Attacker node) └── Windows Server (AD Domain Controller — pending)

Isolation Validation

TestResult
ping 8.8.8.8 from lab deviceSuccess — internet works
ping 10.0.0.1 (home router) from labBlocked — isolation confirmed
ping 10.0.0.16 (Pi wlan0) from labBlocked — isolation confirmed
DNS resolution from labWorking — 8.8.8.8 via Pi NAT

Lessons Learned

Both INPUT and FORWARD rules are required for complete isolation — FORWARD alone allows traffic through even if the Pi itself is unreachable. mDNS does not cross NAT boundaries, which eliminated one potential home network leak vector. Running Suricata on a Pi 4 that is also doing routing under load produced zero packet drops — the Pi 4 is a capable gateway IDS for lab-scale traffic.

Documentation / Sensors & Network
Raspberry Pi Cluster
✓ Completed Category: Sensors · Network Architecture Updated: March 2026

Architecture Overview

Three Raspberry Pi units form both the physical network boundary and the sensor layer of the lab. This is not a supplementary add-on — the Pi cluster is the foundation that enables full network isolation, traffic inspection, and deception-based detection before a single packet reaches a VM.

NodeModelRoleStatus
Pi #1Raspberry Pi 3BWAN Bridge — Wi-Fi NAT gatewayOnline
Pi #2Raspberry Pi 4Suricata IDS — Gateway sensorOnline
Pi #3Raspberry Pi 4Cowrie Honeypot — SSH/Telnet deceptionOnline

Pi #1 — WAN Bridge (Raspberry Pi 3B)

The Pi 3B acts as a custom upstream ISP for the lab — bridging the home Wi-Fi network to the ASUS router WAN port via a double-NAT boundary. This creates complete isolation from the home LAN (10.0.0.0/24) while still allowing internet access inside the lab. The Pi enforces firewall rules at the boundary that prevent any lab device from reaching the home network.

interfaces # Pi 3B network roles wlan0 → 10.0.0.16 # upstream: home Wi-Fi eth0 → 192.168.100.2 # downstream: ASUS WAN port # NAT masquerade — all lab traffic exits via wlan0 iptables -t nat -A POSTROUTING -o wlan0 -j MASQUERADE # Isolation rules — lab devices cannot reach home LAN iptables -I FORWARD 1 -i eth0 -o wlan0 -d 10.0.0.0/24 -j DROP iptables -I INPUT 1 -i eth0 -d 10.0.0.16 -j DROP
The ASUS router sits behind this Pi and operates on its own LAN subnet (192.168.50.0/24), completely unaware of the home network. This is a real-world ISP-style architecture — the Pi is the upstream provider.

Pi #2 — Suricata IDS (Raspberry Pi 4)

A Raspberry Pi 4 runs Suricata in Gateway IDS mode — positioned so that all traffic entering and leaving the lab passes through it. This provides network-level visibility across DNS, HTTP, TLS, flow records, and signature-based alerts before anything reaches the Proxmox VMs.

bash # Validate Suricata config and interface binding sudo suricata -T -c /etc/suricata/suricata.yaml -i wlan0 # Run Suricata as Gateway IDS daemon sudo suricata -c /etc/suricata/suricata.yaml -i wlan0 -D # Update Emerging Threats ruleset sudo suricata-update sudo systemctl restart suricata # Output: /var/log/suricata/eve.json (EVE JSON — SIEM-ready)

A custom logrotate policy was built to protect the SD card from log growth: daily rotation, 7 compressed archives, and automatic Suricata reload. Performance was validated at 0 packet drops under full lab load.

EVE JSON output is the industry standard format for SIEM ingestion. When the Wazuh agent is deployed to this Pi, it will tail eve.json and forward structured alerts directly to the Wazuh Manager on the Ubuntu SOC VM.

Pi #3 — Cowrie Honeypot (Raspberry Pi 4)

A dedicated Raspberry Pi 4 runs Cowrie, a medium-interaction SSH/Telnet honeypot. Any connection attempt to this node triggers full session recording — capturing commands, credentials, malware downloads, and attacker behavior patterns in JSON format.

bash # Cowrie deployed from source with Python 3.11 (compiled from source) # — Pi OS shipped an incompatible version, resolved by building Python git clone https://github.com/cowrie/cowrie cd cowrie pip install --user -r requirements.txt cp etc/cowrie.cfg.dist etc/cowrie.cfg # Cowrie runs as a systemd service for 24/7 persistence sudo systemctl enable cowrie sudo systemctl start cowrie # Captures: brute-force attempts, commands, malware, IPs, tools used # Output: JSON logs → forwarded to SIEM pipeline (planned)

What This Multi-Sensor Layer Means

Combined, the three Pi nodes create a detection pipeline that operates independently of the VM layer. Suricata sees every network packet; Cowrie captures active attacker behavior; the WAN bridge enforces the isolation boundary that keeps the lab safe. This architecture mirrors how enterprise SOC environments deploy out-of-band network taps and deception technology — and it runs on three $40 devices.

Lessons Learned

Suricata must be bound to the correct interface — validating with -T before running live saved significant debugging time. Log rotation is not optional on SD-card-based systems; without it, Suricata will fill the card within days. For Cowrie, Python version conflicts are common on Pi OS — compiling from source is the cleanest fix and demonstrates genuine problem-solving under constraint.

Incident Reports
✓ Active Category: Troubleshooting Logs Updated: March 2026

Real problems encountered during lab build and operation, documented in incident report format. Each entry follows the same structure used in enterprise IT environments: Issue → Diagnostics → Resolution → Verification.

IR-001 Suricata Failing to Bind to Network Interface Resolved
Issue
Suricata started without errors but produced no alerts and logged no traffic. EVE JSON output file remained empty despite network activity confirmed on the Pi.
Diagnostics
Ran config validation to check interface binding:
bash sudo suricata -T -c /etc/suricata/suricata.yaml -i eth0 # Error: eth0 not found — Pi uses wlan0 as upstream interface
Confirmed active interface with ip link show. Interface was wlan0, not eth0. Config file had been set to eth0 based on generic documentation.
Resolution
Updated /etc/suricata/suricata.yaml to bind to wlan0. Restarted service and re-validated with -T flag — no errors. EVE JSON output immediately populated on next test ping.
Verification
Confirmed 0 packet drops in decoder stats. Alerts generating correctly. Ran tail -f /var/log/suricata/eve.json and observed live DNS and flow records during normal lab traffic.
Lesson
Always validate interface names on the actual hardware before deployment. Generic docs assume eth0 — Raspberry Pi OS uses wlan0 for Wi-Fi upstream. The -T (test mode) flag is the correct first diagnostic step for any Suricata binding issue.
IR-002 Cowrie Honeypot Failing to Start — Python Version Conflict Resolved
Issue
Cowrie installation completed without errors but the service failed to start. systemctl status cowrie showed a Python syntax error during import.
Diagnostics
bash python3 --version # Python 3.9.2 — Cowrie requires 3.11+ pip3 install -r requirements.txt # Multiple dependency failures on 3.9
Raspberry Pi OS Bullseye ships Python 3.9 by default. Cowrie's current dependency chain requires 3.11 minimum.
Resolution
Compiled Python 3.11 from source on the Pi. Created a dedicated virtual environment pointing to the compiled binary, then reinstalled all Cowrie dependencies cleanly inside the venv.
Verification
Service started successfully. Confirmed SSH honeypot listening on configured port. Tested with a manual SSH connection attempt — Cowrie captured the session and logged it to JSON output.
Lesson
Default OS Python versions are often behind project requirements. Compiling from source is the clean fix — avoid adding PPAs or third-party repos to production-adjacent systems. Always check requirements.txt before installation.
IR-003 ASUS Router Not Receiving DHCP from Pi 3B WAN Bridge Resolved
Issue
ASUS router WAN port showing "No IP address" after connecting to the Pi 3B eth0. Lab devices had no internet access. Pi appeared to be running correctly.
Diagnostics
Used tcpdump to observe DHCP traffic on the Pi eth0 interface:
bash sudo tcpdump -i eth0 -n port 67 or port 68 # DHCP DISCOVER packets arriving from ASUS # No DHCP OFFER being sent back from Pi
DHCP DISCOVER packets were arriving from the ASUS router but no OFFER was being sent. Traced to isc-dhcp-server not listening on eth0 — the INTERFACES config in /etc/default/isc-dhcp-server was blank.
Resolution
Set INTERFACESv4="eth0" in /etc/default/isc-dhcp-server. Restarted the DHCP service. ASUS WAN immediately received an IP in the 192.168.100.x range.
Verification
ASUS WAN status showed connected with correct IP. Devices on ASUS LAN confirmed internet access via ping 8.8.8.8. Home LAN isolation confirmed — ping 10.0.0.1 blocked as expected.
Lesson
isc-dhcp-server requires explicit interface binding in its defaults file — it won't listen on any interface until told to. tcpdump is the correct tool for diagnosing DHCP failures: if DISCOVER arrives but no OFFER is sent, the problem is always server-side configuration, not the client.
Documentation / Standard Operating Procedures
Standard Operating Procedures
✓ Active Category: SOPs Updated: March 2026

Standardised workflows for common lab operations. Written to be repeatable by anyone with access to the environment — the same standard expected in MSP and enterprise IT documentation.

SOP-101 Provisioning a New VM on Proxmox
Applies to: All new VM deployments  ·  Est. time: 20–30 min
  1. Log into Proxmox web UI at https://<proxmox-ip>:8006
  2. Upload ISO to local storage if not already present: Storage → local → ISO Images → Upload
  3. Click Create VM. Set VM ID, name, and select the ISO
  4. Assign resources: minimum 2 vCPU, 4GB RAM, 40GB disk on local-lvm
  5. Set network bridge to vmbr1 for SOC subnet isolation or vmbr0 for LAN access
  6. Start VM, complete OS installation, set static IP in the appropriate subnet range
  7. Configure NTP synchronisation immediately after OS install
  8. Update asset inventory on index.html and the relevant documentation page
SOP-102 Updating Suricata Rules
Applies to: Pi 4 Suricata IDS  ·  Frequency: Weekly recommended
  1. SSH into the Suricata Pi: ssh pi@<suricata-ip>
  2. Run rule update: sudo suricata-update
  3. Validate config before restart: sudo suricata -T -c /etc/suricata/suricata.yaml -i wlan0
  4. If validation passes, restart: sudo systemctl restart suricata
  5. Confirm service healthy: sudo systemctl status suricata
  6. Tail EVE JSON briefly to confirm alerts still generating: tail -f /var/log/suricata/eve.json
SOP-103 Responding to a Cowrie Honeypot Alert
Applies to: Pi 4 Cowrie Honeypot  ·  Trigger: New session in cowrie.json
  1. Review latest Cowrie JSON log: tail -50 /home/cowrie/cowrie/var/log/cowrie/cowrie.json | python3 -m json.tool
  2. Note attacker IP, timestamp, commands run, and any files attempted for download
  3. Cross-reference IP against Suricata EVE JSON for correlated network events
  4. Check if IP appears in other lab logs (Ubuntu auth.log, Windows Event Viewer)
  5. Document the session: record technique, tools, and any IOCs (Indicators of Compromise)
  6. If a real attack tool was downloaded, capture the hash for future reference
  7. Add any new attacker IPs or patterns to Suricata custom rules if appropriate
SOP-104 Pi 3B WAN Bridge — Restoring Connectivity After Reboot
Applies to: Raspberry Pi 3B  ·  Trigger: Pi rebooted or DHCP lease expired
  1. SSH into Pi 3B from main PC: ssh pi@10.0.0.16
  2. Verify both interfaces are up: ip link show wlan0 && ip link show eth0
  3. Confirm NAT rule is active: sudo iptables -t nat -L -n -v | grep MASQUERADE
  4. If NAT rule missing, re-apply: sudo iptables -t nat -A POSTROUTING -o wlan0 -j MASQUERADE
  5. Confirm DHCP server running: sudo systemctl status isc-dhcp-server
  6. If DHCP stopped: sudo systemctl start isc-dhcp-server
  7. From a lab device, test: ping 8.8.8.8 (internet) and confirm ping 10.0.0.1 is still blocked
